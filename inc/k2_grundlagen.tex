\section{Domain-Driven Design}

\subsection{Definition}

Modelle sind bewusst simplifizierte und strukturierte Darstellungen von Wissen.
Die Methodik des Domain-Driven Design (DDD) stellt drei Regeln auf, welche bei der Wahl eines Modells zu berücksichtigen sind: Modell und Implementierung beeinflussen sich gegenseitig. Das Modell ist die Grundlage der im Entwicklungsprozess verwendeten Terminologie. Das Modell beinhaltet Wissen in einer nutzbaren, aufgearbeiteten Form.
\cite[S. 3-4]{evans}

Fachlichkeiten aus einer Domäne werden mit einer Unbiquitous Language, einer dem Model-basieten Sprache welche Entwickler und Domänen-Experten eine gemeinsame Terminologie vorgibt, ausgedrückt.
\cite[S. 25-27]{evans}

Im Domain-Driven Design (DDD) gewährleistet das Model-Driven Design eine enge Abstimmung zwischen Software-Design und konzeptionellem Domänenmodell. Um eine hohe Übereinstimmung zwischen Modell und Design zu erreichen, ist es fast unerlässlich, innerhalb eines von Software-Werkzeugen unterstützten Modellierungsparadigmas zu arbeiten. Ein ideales Werkzeug sind objektorientierte Programmiersprachen, da Modellkonzepte sich recht gut in objektorientierte Konzepte übersetzen lassen.
\cite[S. 49-51]{evans}

\section{Microservices}

\subsection{Definition}

Der zur Modularisierung eingesetzte Ansatz der Microservices erfüllt verschiedene Grundlegende Vorraussetzungen:
In ihrer Implementierung gibt es keine technologischen Einschränkungen. Die Dienste können unabhängig voneinander überarbeitet und auf ein Zielsystem ausgerollt werden. Weiterhin sollen Build und Deployement automatisiert sein. Sie besitzen einen eigenen Datenhaushalt sowie eigenständige Prozesse auf der ausführenden Infrastruktureinheit. Microservices nutzen verteilte Netzwerkkommunikation, um sich mit anderen Diensten zu verständigen. Transaktionen zeichnen sich durch die ACID-Eigenschaften aus, zu denen Atomizität, Konsistenz, Isolation und Dauerhaftigkeit gehören. Es ist wichtig, dass Daten konsistent bleiben. Deshalb sollten keine Transaktionen über mehrere Services hinweg durchgeführt werden. Um diese Konsistenz zu gewährleisten, können Kompensationstransaktionen eingesetzt werden.
\cite[S. 2]{wolff}
\cite[S. 31-37]{wolff}

\subsection{Architektur von Microservice-Systemen}

Jeder Microservice soll eine eigene fachliche Einheit bilden, die so geschnitten ist, dass für Änderungen oder neue Features nur ein Microservice geändert werden muss. Weiterhin sollen die aus dem DDD bekannten Konzepte der Unbiquitous Language und des Bounded Context verwendet werden.

-DDD: Bounded Context: Jedes Domänenmodell ist nur in bestimmten Grenzen sinnvoll. (Stategic Design) Verschiedene Arten von Kollaboration zwischen Bounded Contexts: Shared Kernel, Customer/Supplier, Conformists, Anti Corruption Layer, Seperate Ways, Open Host Service, Published Language. Es sollte vermieden werden, dass Microservices mehrere Bounded Contexts enthalten.

- Large-Scale Strukturen: System Methapor, Responsibility Layer, Evolving Order.
\cite[S. 44-53]{wolff}

- ???: Jeder Microservice benötigt eine eigene Continous-Delivery-Pipeline. Jeder muss außerdem ein Monitoring haben. Unabhängige Versionskontrolle
